# 跨项目通用映射规则

本文档是从当前“商机数据整合”项目中抽离出来的通用版规则，目标是让其他项目在字段名、sheet 名、文件名不同的情况下，也能复用同一套业务映射逻辑。

## 1. 适用范围

适用于以下类型的数据整合项目：

- 网站注册 / 表单 / 留资数据
- 客服聊天记录
- 400 电话记录
- 订单数据
- 其他包含 `utm_*`、手机号、邮箱、时间、线索状态字段的数据表

不依赖特定文件名，不依赖特定 sheet 名，不依赖当前项目目录结构。

## 2.1 输出形态规则

- 每次输出都应是一个合并后的终极结果表。
- 不建议按来源拆成多个结果 sheet 或多个结果文件。
- 推荐一个工作簿只保留一个主数据 sheet，承载所有来源合并后的标准化结果。
- 如果需要补充统计说明，优先以内嵌备注或附注形式保留，而不是拆出独立 summary 表。

## 3. 复用原则

### 3.1 先做字段适配，再做业务映射

其他项目接入时，先把源字段适配到标准字段语义，再套用业务规则。

例如：

- 源字段 `State`、`State省`、`省份` 都应先对齐成 `province`
- 源字段 `手机号`、`Mobile Phone`、`来电手机号` 都应先对齐成 `mobile Phone`
- 源字段 `广告来源`、`utm_source`、URL 中解析的 `utm_source` 都应先对齐成 `utm_source`

### 3.2 业务规则优先于源表结构

可复用的是“业务口径”，不是当前项目里的具体表头。

也就是说，其他项目里即使：

- 文件名不同
- sheet 名不同
- 表头位置不同
- 字段名不同

只要业务含义相同，都应该映射到同样的标准字段，再走同样的清洗与 channel 映射逻辑。

## 4. 通用清洗规则

### 4.1 account id

- 先转字符串并去首尾空格
- 若值为以下任一内容，则清空：
  - `未查到`
  - `无公司名`

### 4.2 手机号

- 去首尾空格
- 如存在 `15934561712.0` 这类值，先去掉 `.0`
- 去除非数字字符后校验
- 合法口径：
  - 11 位数字
  - 以 `1` 开头
- 不合法则清空

### 4.3 邮箱

- 统一转小写
- 需满足基础邮箱格式
- 不合法则清空

### 4.4 时间字段

适用于：

- `capture date`
- `call date`
- 其他同类业务日期字段

处理规则：

- 合法日期 / 日期时间统一输出 `YYYY-MM-DD`
- 无法识别则清空

### 4.5 call quarter

- 如果来源是 `Business Week` 一类字段，则保留 `W` 前面的季度部分
- 例如：
  - `FY27Q2W02 -> FY27Q2`

## 6. channel_classification / channel 通用规则

### 6.1 输入字段

channel 映射至少应优先使用以下字段：

- `utm_source`
- `utm_content`
- `utm_campaign`

必要时可补充：

- `form_title`
- `module_source`

### 6.2 映射优先级

建议按以下顺序处理：

1. 先匹配显式 channel 规则表
2. 再做来源别名归一
3. 如果仍未命中，则默认：
   - `channel_classification = 自然流量`
   - `channel = 自然流量`

### 6.3 匹配规则建议

- 文本统一转小写
- 去首尾空格
- 连续空格压缩为单空格
- `【包含】` / `【包括】` 按子串匹配
- `【不包含】` 为排除条件
- `【等于】` 为精确匹配
- 当存在更具体来源值时，具体规则优先于宽泛规则

例如：

- `wechat-dsp-xcx` 应优先于 `wechat-dsp`
- `xiaohongshu_xcx` 应优先于 `xiaohongshu`
- `wechat_dellqicai_menu` 可归并到 `wechat_dellqicai`

### 6.4 未命中默认值

如果项目要求 `channel_classification` 和 `channel` 不允许为空，则统一兜底为：

- `channel_classification = 自然流量`
- `channel = 自然流量`

## 7. 其他项目接入时需要单独确认的内容

以下内容通常不能直接照搬，需要项目接入时单独确认：

- 哪个 sheet 是实际数据源
- 表头起始行和列
- 哪些来源字段需要过滤
- 是否需要跨来源去重
- 是否存在额外数据库补充字段
- 是否存在项目专属 `utm_source` / campaign 命名

## 8. 推荐落地方式

其他项目使用本规则时，建议拆成三层：

### 8.1 字段适配层

把不同来源字段统一成标准字段：

- `province`
- `utm_source`
- `mobile Phone`
- `capture date`
- `call type`

### 8.2 清洗层

执行：

- 手机号清洗
- 邮箱清洗
- 日期清洗
- account id 清洗
- call type 统一映射

### 8.3 业务映射层

执行：

- `channel_classification`
- `channel`
- `Campaign Name`
- `Allocaida ID`
- `call quarter`
- 其他业务口径字段

## 8.4 Campaign / Allocaida 派生规则

- 如果项目中 `Campaign Name`、`Allocaida ID` 属于业务归因字段，优先建议按最终 `channel` 统一派生，而不是直接复用不同来源表中的原始值。
- 当前项目的长期口径为：
  - `channel` 为 `douyin`、`toutiao`、`kuaishou`、`xiaohongshu`、`xiaohongshu_xcx`
    - `Campaign Name = FY27Q2_APJC_GC_CN Eshop_Promotion DY/Toutiao/Kuaishou`
    - `Allocaida ID = 3445890`
  - `channel` 为 `wechat-dsp`、`wechat-dsp-xcx`
    - `Campaign Name = FY27Q2_APJC_GC_CN Eshop_Promotion Tencent-WPP`
    - `Allocaida ID = 3445892`
  - 其他所有 `channel`
    - `Campaign Name = FY27Q2_APJC_GC_CN Commerical Membership MA`
    - `Allocaida ID = 3445891`
- 如果其他项目也采用这种做法，建议把这组派生规则独立维护成一层配置，便于后续统一替换。
