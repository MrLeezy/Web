# 通用清洗规则

本文档记录整合终极表时必须长期保持一致的字段清洗规则。

## 1. account id

- 读取后先转字符串并去首尾空格。
- 如果值为以下任一内容，则清空：
  - `未查到`
  - `无公司名`
- 其他值按原样保留。

## 2. capture date

- 统一输出格式：`YYYY-MM-DD`
- 允许输入类型：
  - Excel 日期单元格
  - Python `datetime`
  - 合法字符串日期
- 允许解析的常见字符串格式：
  - `YYYY-MM-DD`
  - `YYYY/MM/DD`
  - `YYYY-MM-DD HH:MM:SS`
  - `YYYY/MM/DD HH:MM:SS`
  - `YYYY-MM-DD HH:MM`
  - `YYYY/MM/DD HH:MM`
- 无法识别的值直接清空。

## 3. call date

- 与 `capture date` 保持同一清洗逻辑。
- 最终统一输出 `YYYY-MM-DD`。
- 无法识别的值直接清空。

## 4. call type

统一映射规则如下：

| 原值 | 输出值 |
| --- | --- |
| `QSRL` | `Leads` |
| `Leads` | `Leads` |
| `Route to CSB` | `Leads` |
| `Route to Dell Solution` | `Leads` |
| `Route to EMC Solution` | `Leads` |
| `SRL` | `Signals` |
| `Signal` | `Signals` |
| `Signal-MRL` | `Signals` |
| `Signals` | `Signals` |
| `SRL(待确定)` | `Signal(待确定)` |
| `Signal(待确定)` | `Signal(待确定)` |
| `非SRL` | `Non-Signals` |
| 其他任意非空值 | `Non-Signals` |
| 空值 | 空 |

## 5. 手机号

- 最终目标：只保留合法中国大陆手机号。
- 规则：
  - 去首尾空格
  - 像 `15934561712.0` 这样的值先去掉 `.0`
  - 再去掉非数字字符
- 校验口径：
  - 必须是 11 位数字
  - 必须以 `1` 开头
- 不满足规则则清空。

## 6. 邮箱

- 邮箱统一转小写。
- 使用基础邮箱格式校验：
  - 必须包含 `@`
  - 必须包含合法域名后缀
- 不满足规则则清空。

## 7. 时间与格式异常值

- 所有时间字段、手机号字段、邮箱字段都执行“格式不正确直接清空”的口径。
- 不保留 raw 字段，不额外新增异常列。
- 原始值仅存在于源文件，不回写入终极表。

## 8. call quarter

- 对于来自 `Business_Week` / `Business Week` 的来源：
  - 取 `W` 之前的内容
  - 例如 `FY27Q2W02 -> FY27Q2`
- 如果原值本身已是季度字段，则直接保留。

## 8.1 call wk

- `call wk` 统一保留为 `FYxxQxWnn` 这种两位周数格式。
- 如果输入是 `FY27Q2W5` 这种单数字周数，统一标准化为 `FY27Q2W05`。
- 如果输入本身已经是 `FY27Q2W05`，则保持不变。
- 不符合该模式的值先按原样保留。

## 9. channel 相关字段

- `channel_classification` 和 `channel` 不允许手工推断。
- 必须严格调用 [`渠道映射.xlsx`](/Users/lizhongyu/程序/团队协作系统/data/opportunity-master/渠道映射.xlsx) 规则生成。
- 若未命中任何规则，默认输出：
  - `channel_classification = 自然流量`
  - `channel = 自然流量`

## 10. Campaign Name / Allocaida ID

- `Campaign Name` 和 `Allocaida ID` 不再从原始来源表直接取值。
- 必须在 `channel` 生成完成后，按最终 `channel` 统一派生。
- 当前长期规则如下：
  - `channel` 为 `douyin`、`toutiao`、`kuaishou`、`xiaohongshu`、`xiaohongshu_xcx`
    - `Campaign Name = FY27Q2_APJC_GC_CN Eshop_Promotion DY/Toutiao/Kuaishou`
    - `Allocaida ID = 3445890`
  - `channel` 为 `wechat-dsp`、`wechat-dsp-xcx`
    - `Campaign Name = FY27Q2_APJC_GC_CN Eshop_Promotion Tencent-WPP`
    - `Allocaida ID = 3445892`
  - 其他所有 `channel`
    - `Campaign Name = FY27Q2_APJC_GC_CN Commerical Membership MA`
    - `Allocaida ID = 3445891`
- 后续如果业务口径调整，只改这一层派生规则，不回退到按来源表头直接取值。

## 11. SFDC Signal 匹配覆写

- 如果工作区同时上传了 `SFDC Signal` 映射文件，终极表会在常规清洗后，额外按 `手机号` 和 `邮箱` 做一次匹配。
- 匹配键使用标准化后的值：
  - 手机号：去非数字，支持去掉前缀 `86`，最终要求是 11 位中国大陆手机号
  - 邮箱：转小写后按基础邮箱格式校验
- 如果整合表当前 `call type` 已经是 `Leads`，即使命中 SFDC 文件，也保持 `Leads` 不变。
- 对于其他所有 `call type`，只要手机号或邮箱任一命中 SFDC 文件，就统一覆写为 `Signals`。
- 当前这层逻辑只把 SFDC 文件当作“命中名单”使用，不再读取其中的 `call_type` 值。
- 如果没有命中，终极表保留原始清洗后的 `call type`。
