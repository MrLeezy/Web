# 插件交付与合并检查清单

这份清单用于两种场景：

- 你把任务发给其他 AI 时，要求它们按这个清单自检
- 它们交付后，你或我按这个清单快速判断能不能并回来

## 1. 开发前必须确认

- 这是在当前仓库里开发，不是单独新建项目
- 已阅读 [plugin-spec.md](/Users/lizhongyu/程序/团队协作系统/docs/plugin-spec.md)
- 已确定插件 ID，并确保不与现有插件重复
- 已确认这是“新增插件”还是“改造现有插件”
- 已确认不需要引入新框架、新构建工具、新服务

## 2. 代码结构检查

如果是新增插件，至少应看到这些文件或等价实现：

```text
plugins/<plugin-id>.html
plugins/<plugin-id>.css
plugins/<plugin-id>.js
server.js
```

如果需要注册卡片，还应有：

```text
data/plugin-cards.json
```

如果需要持久化数据，路径应该类似：

```text
data/<plugin-id>/
private/<plugin-id>/
```

不应出现：

- 一个新的独立前端项目目录
- 新的 `package.json` 子项目
- 新的 dev server
- 新的 build 脚本体系

## 3. 插件注册检查

- `DEFAULT_PLUGIN_CARDS` 已新增对应卡片
- 卡片 `id`、文件名、API 命名空间一致
- `pageUrl` 指向 `/plugins/<plugin-id>.html`
- 首页 `PLUGINS` 可以看到该卡片
- 卡片点击后能打开对应页面

## 4. 接口检查

- 存在 `GET /api/plugins/<plugin-id>/health`
- 业务接口都挂在 `/api/plugins/<plugin-id>/...`
- 成功返回 JSON
- 失败也返回 JSON
- 错误文案是可读的，不只是原始异常
- 有输入校验，不会直接信任前端传参

## 5. 页面检查

- 页面能正常打开
- CSS 与 JS 正常加载
- 有返回首页入口
- 有插件标题、用途说明
- 有主操作区
- 有状态提示区
- 有加载态、成功态、失败态
- 窄屏下仍能操作，不严重错位
- 页面不会因为接口失败而白屏

## 6. 数据与安全检查

- 敏感信息没有写进前端
- 数据写入 `data/` 或 `private/` 的插件专属目录
- 没有让前端直接读取 `/data` 或 `/private`
- 文件名和路径有做清洗
- 上传文件类型有限制
- 不存在明显的路径穿越风险

## 7. 回归检查

- 没有改现有插件 ID
- 没有改其他插件 API 前缀
- 没有重写整个 `PLUGINS` 渲染逻辑
- 没有影响首页其他模块
- 没有影响后台已有配置能力
- 没有影响现有插件主流程

## 8. 必须向你回传的信息

其他 AI 交付时，至少必须回传以下内容：

- 本次实现概述
- 修改文件清单
- 新增或修改的接口列表
- 数据目录说明
- 手动验证步骤
- 已知限制
- 风险说明

可直接要求它按这个格式回：

```md
本次实现概述：
- ...

修改文件：
- ...

接口变更：
- ...

数据目录：
- ...

验证步骤：
1. ...
2. ...
3. ...

已知限制：
- ...

风险说明：
- ...
```

## 9. 你可以直接发给其他 AI 的验收要求

```md
请在交付前按以下标准自检并在回复中明确说明：

1. 是否完全在当前仓库内实现，没有新建独立项目
2. 是否遵循 `docs/plugin-spec.md`
3. 是否列出了所有修改文件
4. 是否列出了所有接口变更
5. 是否说明了数据目录
6. 是否给出了手动验证步骤
7. 是否说明了已知限制和风险
8. 是否确认不会影响现有项目其他功能
```

## 10. 最终建议

如果你的目标是“尽量少返工、一次合并成功”，最稳妥的工作流是：

1. 先把 [plugin-spec.md](/Users/lizhongyu/程序/团队协作系统/docs/plugin-spec.md) 和 [plugin-task-template.md](/Users/lizhongyu/程序/团队协作系统/docs/plugin-task-template.md) 一起发给其他 AI
2. 要求它按本清单格式交付
3. 再由我做最终合并、校验和收口

这样最容易避免“他们做的是另一个项目，不是你这个项目里的插件”这种返工。
