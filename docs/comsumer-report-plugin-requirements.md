# Consumer 数据分析插件需求文档

## 1. 文档目的

本文档用于指导其他 AI 在当前仓库内新增一个“Consumer 数据分析插件”。目标不是新建独立项目，而是在当前项目中实现一个可配置、可上传数据、可生成多张分析底表的内置插件。

注意：

- 仓库现有目录名使用的是 `comsumer`，这是当前事实路径，实施时应继续沿用现有路径，避免破坏已有数据文件引用。
- 面向用户展示的插件名称建议使用 `Consumer 数据分析`，但代码路径、数据目录可以继续使用 `comsumer`。

## 2. 业务目标

当前 Consumer 数据分析报告主要依赖人工维护多个 Excel 表和手工归类逻辑，效率低、容易出错。目标是做成一个可配置、可重复执行、可更新的插件，核心能力包括：

- 维护渠道字典和分析字典，而不是长期手工改 Excel
- 上传线下表、读取数据库结果、做统一清洗和映射
- 生成并更新三张核心底表：
  - `UserRawdata_merged.xlsx`
  - `DemandRawdata.xlsx`
  - `ProductRawdata.xlsx`
- 支持按 `Year / Quarter / Month / Week` 进行数据更新
- 对于周报类底表采用“按周期 upsert”
- 对于日报需求表采用“按日期累积更新”
- 对于产品表采用“每次覆盖刷新”

## 3. 插件范围

本插件至少包含以下功能模块：

1. 配置中心
2. 周维度用户底表生成与更新
3. 日维度需求底表生成与更新
4. 产品底表生成与覆盖
5. 数据导入与执行日志

不在本期强制范围内：

- 通用 BI 看板
- 自动数据库建模
- 对现有非 Consumer 插件的重构

## 5. 核心统一规则

这是整个插件最重要的抽象。很多业务步骤虽然看起来繁琐，但本质上反复在做同几件事：

1. 时间维度标准化
2. UTM 映射到唯一标识符
3. 原始明细打标签
4. 按维度聚合写入目标表

为避免逻辑重复，其他 AI 必须按“公共规则引擎”实现，而不是每个数据源各写一套。

### 5.1 时间维度标准化规则

所有数据进入系统后，尽量先转成统一字段：

- `Year`
- `Quarter`
- `Month`
- `Week`
- `Day`（仅对日级分析数据需要）

不同数据源可能只有日期字段，或者只有 `Fiscal_Quarter` / `Fiscal_Week`，插件必须负责统一转换。

### 5.2 唯一渠道映射规则

统一映射链路：

1. 从原始数据中抽取 UTM 相关字段
2. 标准化为：
   - `utm_source`
   - `utm_campaign`
   - `utm_term`
3. 三字段合并后，在 `配置表.xlsx` 的 `UTM` sheet 中找到对应关系
4. 得到：
   - `Channel`
   - `Source`
   - `Term`
   - `来源合并`
   - `唯一标识符`
5. 再写入目标底表

若原始数据不是标准 UTM 字段，而是拼在 URL 或别名字段中，插件必须先拆解或归一化再映射。

### 5.3 唯一标识符规则

`唯一标识符` 来源于 `配置表.xlsx -> 唯一渠道数据分布` sheet 的 E 列。

周表中每一行的业务主键建议使用：

`Year + Quarter + Week + 唯一标识符`

对应当前表中的：

- E 列 `当周数据匹配`

### 5.4 段位（Segment）规则

需求表中的段位当前至少有：

- `Con ALW`
- `Con NonALW`

其中：

- `Con ALW` 表示匹配到外星人相关产品/关键词的有效需求
- `Con NonALW` 表示有效需求但未匹配到外星人相关条件
- `无需求` 不进入有效需求统计

## 7. UserRawdata_merged 周表需求

## 7.1 总体规则

- 用户在插件前端先选择：
  - `Year`
  - `Quarter`
  - `Month`
  - `Week`
- 若周表中已存在该周期对应行，则执行更新
- 若不存在，则基于“唯一渠道数据分布”生成该周的基准行后再写入

## 7.2 行生成规则

一周的标准行集来自：

- `配置表.xlsx -> 唯一渠道数据分布`

也就是说，若字典中有 20 个唯一渠道，则每个周维度至少应生成这 20 条基础行。

基础字段生成：

- A-D：前端选择的时间
- G-J：从标准渠道字典带出
- E：`Year + Quarter + Week + 唯一标识符`
