# 给其他 AI 的标准任务模版

这份文档用于你把插件或功能任务直接分发给其他 AI。目标是让他们“直接在当前仓库里开发”，避免先做独立项目、后续再返工合并。

配套规范请同时附上：

- [plugin-spec.md](/Users/lizhongyu/程序/团队协作系统/docs/plugin-spec.md)

## 1. 通用总约束

每次发任务时，建议先附带这段固定约束，不要删：

```md
你是在当前仓库内开发功能，不是新建独立项目。请直接在现有项目里实现，并严格遵循 `docs/plugin-spec.md` 的约束。当前项目的插件是内置式结构，前端文件放在 `plugins/`，页面入口为 `/plugins/<plugin-id>.html`，后端接口统一挂在 `/api/plugins/<plugin-id>/...`，并在 `server.js` 的 `DEFAULT_PLUGIN_CARDS` 中注册卡片。不要引入新的前端框架、打包器、第二套服务，不要把功能做成独立项目。所有运行数据放在 `data/` 或 `private/` 下的插件专属目录中，错误统一返回 JSON。若实现方式与当前仓库结构冲突，优先服从当前仓库结构，不要自行抽象成新工程。
```

## 2. 新增插件任务模版

适用于：让其他 AI 从零新增一个插件。

```md
请在当前仓库内新增一个内置插件，名称：`<插件名称>`，插件 ID：`<plugin-id>`。

任务目标：
- `<写清这个插件要解决什么问题>`
- `<写清主流程，例如上传文件、处理数据、查看结果、下载结果等>`

强约束：
- 必须直接在当前仓库内实现，不能新建独立项目
- 必须遵循 `docs/plugin-spec.md`
- 前端页面文件放在 `plugins/`
- 页面入口必须是 `/plugins/<plugin-id>.html`
- 后端接口统一挂在 `/api/plugins/<plugin-id>/...`
- 需要在 `server.js` 的 `DEFAULT_PLUGIN_CARDS` 中注册插件卡片
- 如需持久化数据，只能写入 `data/<plugin-id>/` 或 `private/<plugin-id>/`
- 不要引入 React/Vue/Vite/Webpack/TypeScript 构建链
- 不要修改其他现有插件的 ID、接口命名空间或主流程

功能要求：
- 页面要有返回首页入口
- 页面要有标题、摘要、主操作区、状态提示区
- 至少提供 `GET /api/plugins/<plugin-id>/health`
- 主流程失败时返回 JSON 错误，前端要能展示错误
- 页面要有加载态、成功态、失败态

交付要求：
- 直接给出修改后的代码
- 列出修改文件清单
- 列出新增/修改接口
- 说明数据目录
- 给出手动验证步骤
- 给出已知限制或未完成项

验收标准：
- 首页 PLUGINS 能看到卡片
- 点击卡片能打开插件页
- 主流程能跑通
- 出错时不会白屏
- 不影响现有项目其他功能
```

## 3. 改造现有插件任务模版

适用于：让其他 AI 在现有插件上追加功能，而不是新做一个插件。

```md
请在当前仓库里改造现有插件：`<plugin-id>`。

本次目标：
- `<功能点 1>`
- `<功能点 2>`

修改范围约束：
- 只允许修改与 `<plugin-id>` 直接相关的文件，以及必要的 `server.js` 插件路由部分
- 不要重构整个插件系统
- 不要改动其他插件的行为
- 必须遵循 `docs/plugin-spec.md`

实现要求：
- 保持现有插件入口不变
- 保持现有 API 命名空间不变
- 新增接口也必须挂在 `/api/plugins/<plugin-id>/...`
- 保持现有页面可用，不要因为新增功能导致原主流程失效
- 所有异常返回 JSON，并在前端显示为可读提示

交付时请输出：
- 修改文件列表
- 变更点摘要
- 新增/调整的接口
- 如何验证原功能未回归
- 已知风险
```

## 4. 做普通功能但不做独立插件的任务模版

适用于：只是做某个功能，不一定要形成新的插件卡片。

```md
请直接在当前仓库里实现以下功能，不要新建独立项目，也不要擅自抽象成新的插件系统：

需求：
- `<写具体功能>`

约束：
- 必须优先复用当前项目现有结构
- 若需要新增页面或接口，命名必须与现有代码风格一致
- 不要引入新的前端框架、构建工具、第二套服务
- 不要为了单个需求重构无关模块
- 如果这个功能必须以插件形式存在，必须遵循 `docs/plugin-spec.md`

交付要求：
- 直接给出代码修改
- 标明修改文件
- 标明验证步骤
- 标明是否影响已有功能
```

## 6. 最推荐的实际发送方式

你后面给其他 AI 发任务时，可以直接采用下面组合：

1. 先贴“通用总约束”
2. 再贴一种具体任务模版
3. 最后补充你的业务需求细节

推荐顺序：

```md
请先阅读并严格遵循 `docs/plugin-spec.md`。

<粘贴通用总约束>

<粘贴新增插件 / 改造插件 / 普通功能任务模版>

补充业务信息：
- ...
- ...
```

## 7. 一句最短版

如果你赶时间，至少发这一句：

```md
请直接在当前仓库内实现，不要新建独立项目；所有插件和功能实现必须服从 `docs/plugin-spec.md`，并按“修改文件、接口、数据目录、验证步骤、已知限制”格式交付。
```
